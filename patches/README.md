# Vendor patches

Local edits to third-party packages live here. Each patch is generated by pnpm
and applied automatically on `pnpm install`.

This directory is intentionally tracked even when empty ‚Äî the workflow below
expects it to exist.

## When to patch

Patch a dependency when:

- An upstream bug blocks us and the fix won't land soon (or won't be backported
  to the version we use), AND
- The fix is small enough to maintain in-repo against future package updates,
  AND
- A runtime workaround (config flag, wrapper, monkey-patch) would be uglier or
  more fragile than the patch itself.

Always link to the upstream issue/PR in the patch header comment, and write
removal criteria so the next person knows when it's safe to drop.

## Creating a patch

```bash
# 1. Open a scratch copy of the package for editing.
pnpm patch <pkg>@<version>
# pnpm prints a path like /tmp/<random>; open that directory and edit files.

# 2. Once edits look good, commit them back to a patch file.
pnpm patch-commit /tmp/<random>
# This writes `patches/<pkg>@<version>.patch` and re-runs install so the
# patched version is now in node_modules.
```

`pnpm patch-commit` updates three places automatically:

- `patches/<pkg>@<version>.patch` ‚Äî the diff itself.
- `package.json` ‚Üí `pnpm.patchedDependencies` ‚Äî entry mapping package to patch file.
- `pnpm-workspace.yaml` ‚Üí `patchedDependencies` ‚Äî same mapping (kept in sync by pnpm).

Both registry entries are required. If only one is present, the patch silently
fails to apply on `pnpm install`.

## Verifying a patch is applied

pnpm installs patched packages into a content-addressable directory keyed on
the patch hash:

```bash
ls node_modules/.pnpm/ | grep '<pkg>@<version>_patch_hash='
```

If you see exactly one such directory and `node_modules/<pkg>` resolves there,
the patch is live. If you see multiple, they're stale ‚Äî `pnpm prune` cleans
them up.

## Updating a patch

Editing `patches/<pkg>@<version>.patch` by hand works for one-line tweaks (the
patch is just a unified diff), then run `pnpm install` to reapply. For larger
edits, redo the `pnpm patch` flow against the same package ‚Äî `pnpm patch-commit`
will overwrite the existing file.

## Removing a patch

```bash
# 1. Delete the patch file.
rm patches/<pkg>@<version>.patch

# 2. Strip the registry entries.
#    - package.json: remove the line from pnpm.patchedDependencies.
#    - pnpm-workspace.yaml: remove the line from patchedDependencies.
#    - pnpm-lock.yaml: remove the patchedDependencies block at the top, and
#      strip "(patch_hash=...)" suffixes from playwright-core references.
#      pnpm install will refuse to run until the lockfile is consistent.

# 3. Reinstall.
pnpm install

# 4. Clean orphan content-addressable dirs.
pnpm prune
```

## Conventions

- One file per patched package, named `<pkg>@<version>.patch`.
- Patch header (inside the diff, as comments in the changed source) must explain:
  - The upstream bug being fixed (with links).
  - What the patch does, in plain language.
  - Removal criteria ‚Äî what would have to happen upstream for us to drop this.

A patch without removal criteria becomes permanent dead weight. Always write them.
